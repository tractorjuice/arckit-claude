# Quality Checklist

Quality checks to verify before writing any ArcKit artifact to file.
Run all **Common Checks** plus the relevant **Per-Type Checks** section for the artifact being generated.
Fix any failures before proceeding to write.

## Common Checks

All artifacts must pass these 10 checks:

1. **Document Control complete** -- All 14 standard fields populated (Document ID, Document Type, Project, Classification, Status, Version, Created Date, Last Modified, Review Cycle, Next Review Date, Owner, Reviewed By, Approved By, Distribution).
2. **Document ID convention** -- Follows `ARC-{PROJECT_ID}-{TYPE}-v{VERSION}` pattern (e.g., `ARC-001-REQ-v1.0`). Multi-instance types include a sequence number (e.g., `ARC-001-ADR-001-v1.0`).
3. **No placeholder text** -- No remaining `[PLACEHOLDER]`, `TODO`, `TBD`, `XXX`, or `YYYY-MM-DD` tokens. All fields contain real values.
4. **Classification set** -- One of: PUBLIC, OFFICIAL, OFFICIAL-SENSITIVE, SECRET.
5. **Status set** -- One of: DRAFT, IN_REVIEW, APPROVED, PUBLISHED, SUPERSEDED, ARCHIVED.
6. **Revision History present** -- Table with columns: Version, Date, Author, Changes, Approved By, Approval Date. At least one row for the initial version.
7. **Generation footer present** -- Standard footer at end of document with Generated by, Generated on, ArcKit Version, Project, and Model fields.
8. **Clean Markdown** -- No broken tables, unclosed code blocks, malformed links, or rendering issues.
9. **Consistent tables** -- All tables have header separators, aligned columns, and no missing cells.
10. **Proper heading hierarchy** -- No skipped heading levels (e.g., jumping from `##` to `####`). Logical document structure maintained.

## Per-Type Checks

### ADMP -- ADM Cycle Plan

- All ADM phases (Preliminary through Phase H and Architecture Change Management) accounted for
- Phase descriptions include purpose, inputs, outputs, and key activities
- Stakeholder mapping links each phase to responsible roles
- Tailoring rationale documented for any phases modified or skipped
- ADM cycle iterations clearly identified (initial, iterative refinement)
- Phase sequencing and dependencies between phases documented
- Mermaid ADM cycle diagram present with phase colour-coding (if included)
- Traceability: each work product references its originating ADM phase

### BPCM -- Business Process Model

- Business capability hierarchy with at least 3 levels of detail
- Value streams identified and linked to capabilities
- Process flow descriptions include trigger, actors, inputs, outputs, and outcomes
- Mermaid flowchart or swimlane diagram present for key processes
- Process-to-capability mapping table included
- RACI assignment for critical process steps
- Traceability: processes linked to strategic objectives and architecture principles (PRIN)

### APP -- Application Inventory

- Each application has a unique ID, name, owner, lifecycle status, and hosting model
- Application-to-capability mapping is present
- Strategic fit, technical health, business criticality, and cost dimensions are assessed
- Integration and dependency information is included
- Technology stack, data stores, and operational ownership are documented
- Traceability: applications reference BPCM capabilities and ADMP scope

### APPR -- Application Rationalisation

- Keep/merge/replace/retire decisions are recorded for every in-scope application
- Each decision includes rationale, risk, dependency, owner, timeline, and target platform where relevant
- Portfolio-level benefit and cost assumptions are documented
- Implementation sequencing is defined
- Traceability: rationalisation decisions reference APP application IDs and BPCM capabilities
- Exception register: any deviations formally recorded with justification and approval
- Applicability matrix: principles mapped to architecture domains (B/D/A/T)
- Stakeholder ownership assigned per principle
- Traceability: register entries link to APP source documents

### GAPA -- Gap Analysis (Phase E)

- Capability gap matrix present with at least 5 capabilities
- Each gap includes: current maturity, target maturity, gap size, urgency, severity score
- Severity scoring methodology documented (Size × Urgency matrix)
- Gap heatmap Mermaid quadrant chart present with gaps plotted
- Workstream mapping contains at least 2 workstreams
- Each workstream includes: gaps addressed, dependencies, duration, resource estimates
- Workstream dependency diagram (Mermaid flowchart) present
- Gap-to-risk mapping table present with risk impact ratings
- Assumptions and constraints documented
- Traceability: gaps linked to BPCM source capabilities, workstreams linked to strategic themes (STRAT)

### TRANS -- Transition Architecture

- At least 2 transition architectures defined (not counting baseline and target)
- At least 4 work packages defined with scope, deliverables, dependencies, resources, and timeline
- Each work package has measurable acceptance criteria (minimum 2 per work package)
- Work package dependency diagram (Mermaid flowchart) present and syntactically valid — no `<br/>` in edge labels
- Resource plan table present with FTE allocations per work package and resource type
- Risk register contains at least 3 entries with contingency plans and trigger conditions
- Every GAPA workstream is addressed by at least one work package
- Transition overview table present with wave descriptions, durations, and investment estimates
- Traceability: GAPA gaps → work packages, work packages → transition architectures, work packages → ROAD themes, APPR decisions → work packages

### ACHG -- Architecture Change Request

- Change type classified: adaptive, planned, unplanned
- Impact assessment covers all affected architecture domains (B/D/A/T)
- Scope and rationale documented
- Risk assessment with likelihood and impact
- Approval status and decision trail recorded
- Back-out plan or rollback strategy included for high-risk changes
- Traceability: change references affected APP principles and REPO entries

### BORD -- Architecture Board Charter

- Board charter present with Purpose, Scope, and Authority sections fully populated
- Membership table lists at least 3 roles with vote types (Voting/Observer)
- Quorum requirements explicitly defined
- Decision framework includes voting model table with at least 3 models (Consensus, Majority, Chair)
- Escalation path Mermaid flowchart present (Board → Programme Board → Steering Committee → Executive Board)
- Decision register template present with all required columns
- Compliance scorecard covers at least 3 architecture domains with current/target/gap/priority
- Exception process defines all 5 steps (Submit → Assess → Review → Decide → Appeal) with owners and timelines
- Review cadence defines at least 2 forums with frequency and agenda items
- Traceability: links to PRIN (principles) and at least one other source document (ADMP, APP, GAPA, or ACHG)

### REPO -- Architecture Repository

- Repository structure documented: governance, standards, lessons learned, delivery, requirements
- Artefact inventory with classification and storage location
- Access control and retention policies defined
- Mermaid repository structure diagram present (if included)
- Versioning and change control process documented
- Search and retrieval capabilities described
- Traceability: repository entries reference originating ADM phase and document type
